# PHASE_I_SHOWDOWN_SPEC.md (V1)

> 项目：Holdem Web  
> 范围：摊牌可解释性（牌型文案 + 分池明细 + 赢家摘要）  
> 执行模式：Spec First（先 Spec、后实现、再验收）

## 1. 背景

当前结算阶段只展示筹码飞行动画和简单 toast，缺少“为什么赢”的解释信息。

## 2. 目标（DoD）

1. 在摊牌阶段展示每位摊牌玩家的牌型文案。
2. 展示主池/边池的派奖明细与赢家名单。
3. 提供赢家摘要，帮助玩家快速理解结算结果。
4. 不引入规则分叉（规则仍由 engine 决定）。

## 3. 设计原则

1. 结算数据来源于事件流（`SHOWDOWN_REVEAL`、`POT_AWARDED`）。
2. 牌型文案可由前端复用 `evaluate7` 推导，但仅用于展示。
3. 展示层与规则层解耦：UI 不参与胜负判定。

## 4. 分阶段实施

## I1 - 结果面板基础（本轮）

交付：

1. 新增 `ShowdownPanel` 组件。
2. 在 TableScene 聚合 `SHOWDOWN_REVEAL` 和 `POT_AWARDED` 事件。
3. 面板展示：摊牌玩家 + 牌型文案 + 池子派奖明细 + 赢家摘要。

验收：

1. 能在 showdown 手看到牌型文案。
2. 多池时能看到主池/边池派奖明细。
3. `npm test`、`npm run build` 通过。

## I2 - 节奏与可读性打磨

交付：

1. 面板出现/消失时机与动画节奏协调（淡入/淡出）。
2. 结果面板居中展示并放大，小屏布局与遮挡优化。

验收：

1. 面板不会遮挡关键操作区。
2. 多设备下可读性稳定。

## I3 - 回归与文档收口

交付：

1. 补齐格式化工具与展示文案单测。
2. 跑通回归门禁并回填文档。

验收：

1. `npm run test:regression`、`npm test`、`npm run build` 通过。
2. `GAME_SPEC.md` 与 `日志.md` 完整回填。

## 5. 测试矩阵

1. `I1-T1`：牌型文案映射正确（如四条/高牌）。
2. `I1-T2`：主池/边池文案格式化正确。
3. `I2-T1`：移动端下结果面板不遮挡底部操作区。
4. `I3-T1`：回归门禁全部通过。
