# PHASE_H_BETTING_UI_SPEC.md (V1)

> 项目：Holdem Web  
> 范围：Hero 下注交互增强（金额输入 / 全下入口 / 快捷档位）  
> 执行模式：Spec First（先 Spec、后实现、再验收）

## 1. 背景与问题

当前按钮区存在可玩性缺口：

1. Hero 当前仅能执行“最小加注”，无法选择目标加注额。
2. 缺少全下入口，无法直接触发常见德州动作。
3. 下注交互缺少快捷金额档位与边界提示。

## 2. 目标（DoD）

1. Hero 可在合法区间内选择 `amountTo` 并执行 `BET/RAISE`。
2. Hero 可一键执行 `ALL_IN`（仅在合法时可用）。
3. 输入越界时自动夹取到合法区间并给出可读提示。
4. 不破坏现有规则真值（仍由 engine 判定合法性）。

## 3. 设计原则

1. Engine 继续保持纯规则，不存 UI 输入态。
2. Store 负责把 UI 输入转换为合法动作，并做保护性夹取。
3. UI 只消费合法区间，不自行推导规则。
4. 所有关键路径必须可由单测覆盖（Fake Timers + store 行为测试）。

## 4. 分阶段实施

## H1 - 可控加注金额（本轮）

交付：

1. `legalActions` 暴露 `maxRaiseTo`。
2. ActionPanel 增加加注滑杆与数值输入。
3. `heroRaise(amountTo?)` 支持传入目标金额并在 store 侧夹取到合法区间。

验收：

1. 输入超上限会按上限执行。
2. 输入低于下限会按下限执行。
3. `npm test`、`npm run build` 通过。

## H2 - 全下动作接入

交付：

1. ViewModel 暴露 `canAllIn / allInTo`。
2. ActionPanel 增加“全下”按钮。
3. Store 新增 `heroAllIn()`，合法时派发 `ALL_IN`。

验收：

1. 全下按钮仅在合法时可用。
2. 全下后动作回显与筹码流转正确。

## H3 - 快捷档位与提示

交付：

1. 增加快捷金额档位（例如：最小、1/2 池、2/3 池、底池、全下）。
2. 输入夹取时给出轻量提示（toast）。

验收：

1. 快捷档位金额始终落在合法区间。
2. 不出现“可点按钮但动作非法”。

## H4 - 回归与文档收口

交付：

1. store 回归测试补齐（H1/H2/H3）。
2. 文档与日志回填，更新 `GAME_SPEC.md`。

验收：

1. `npm test` 全绿。
2. `npm run build` 通过。

## 5. 测试矩阵

1. `H1-T1`：`heroRaise(max + delta)` 被夹取到 `maxRaiseTo`。
2. `H1-T2`：`heroRaise(min - delta)` 被夹取到 `minRaiseTo`。
3. `H2-T1`：有 `ALL_IN` 合法动作时按钮可用并成功派发。
4. `H2-T2`：非 Hero 回合或动作锁定时全下按钮不可用。
5. `H3-T1`：快捷档位金额被正确夹取且可执行。
6. `H4-T1`：现有规则回归矩阵继续全通过。

## 6. 风险与防护

风险：

1. UI 输入态与真实合法区间不同步。
2. 金额输入抖动导致用户误触。
3. 新按钮导致小屏遮挡回归。

防护：

1. 合法区间统一来自 `getLegalActions` 映射，不在 UI 重算。
2. 提交前 store 再做一次夹取保护。
3. 增加移动端快速验收截图与按钮区域复核。
